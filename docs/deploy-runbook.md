# מדריך פריסה והפעלה (Deploy Runbook)

מדריך זה מיועד למנהלי מערכת ומפתחים לצורך הקמה, הפעלה מקומית ופריסה של אתר **בית לחתול** לסביבת ייצור.

## 1. מבנה התיקייה בפרויקט

```text
├── .next/                  # תיקיית הבנייה של Next.js
├── app/                    # קוד האפליקציה (Next.js App Router)
│   ├── admin/              # דאשבורד מנהל המערכת
│   ├── adopt/              # שאלון התאמה ותהליך אימוץ
│   ├── api/                # נקודות קצה של API (כולל ניתוב מדיה מאובטח)
│   ├── cats/               # קטלוג החתולים ועמודי פירוט
│   └── publish/            # מערכת העלאת חתולים וניהול מודעות מוסר
├── components/             # רכיבי UI משותפים וממודרים לפי נושאים
├── content/
│   └── he/
│       └── ui.json         # קובץ המחרוזות הראשי (כל תוכן האתר בעברית)
├── docs/                   # תיעוד המערכת וצילומי מסך
├── emails/                 # תבניות אימייל טרנזקציונליות (React Email)
├── lib/                    # פונקציות עזר, קבועים ומחלקות שירות (Supabase client וכו׳)
├── public/                 # קבצים סטטיים (לוגו, וידאו רקע וכו׳)
├── scripts/                # סקריפטים להזנת נתונים (Seeding) ובדיקות
└── supabase/               # קונפיגורציית Supabase מקומית (מיגרוציות ו-RLS)
```

---

## 2. הקמה והפעלה מקומית

לצורך הפעלת האתר בסביבה מקומית, יש לוודא שמותקנים במחשבכם:
* **Node.js** (גרסה 18 ומעלה)
* **Docker Desktop** (הכרחי להפעלת Supabase המקומית)

### שלב א': התקנת תלויות
בצעו את הפקודה הבאה בתיקיית השורש:
```bash
npm install
```

### שלב ב': הפעלת Supabase מקומית
ודאו שתוכנת Docker פתוחה, והריצו:
```bash
npx supabase start
```
פקודה זו תקים ותפעיל את כל הקונטיינרים הדרושים (בסיס נתונים PostgreSQL, שירות אחסון קבצים S3, ממשק ניהול Studio וכו׳).

### שלב ג': הזנת נתוני בדיקה (Seed)
כדי לאכלס את מסד הנתונים בחתולי הדגמה, משתמשים ולקבוע הגדרות RLS תקינות, הריצו את סקריפט הזנת הנתונים:
```bash
node scripts/seed-data.mjs
```
זה ייצור:
* אדמין מערכת (`admin@example.com` / `password123`)
* שני מוסרים מאושרים (`publisher1@example.com` / `publisher2@example.com`)
* 12 חתולים מפורסמים עם תמונות וסרטוני הדגמה
* מנגנוני אבטחה ואימות קבצים

### שלב ד': בנייה והפעלה של האפליקציה

לבניית גרסת ייצור מקומית (Production build):
```bash
npm run build
```

להפעלת שרת הייצור המקומי בפורט `3000`:
```bash
npm run start
```
כעת האתר זמין בכתובת [http://localhost:3000](http://localhost:3000).

## 3. הרצת בדיקות ואיכות קוד (Quality Gate)

לפני כל פריסה לייצור, יש לוודא שכל הבדיקות עוברות בהצלחה:
```bash
npm run gate
```
פקודה זו מריצה באופן אוטומטי:
1. בדיקת תקינות מוסכמות קוד (`check-conventions.mjs`) - מוודאת שאין מחרוזות בעברית קשות בתוך ה-JSX ואין שימוש במחלקות כיוון פיזיות (`left`, `right`).
2. בדיקת סוגים של TypeScript (`tsc --noEmit`).
3. בדיקות איכות קוד ופורמט (ESLint).
4. הרצת בדיקות יחידה של השרת ותבניות המייל (`vitest`).

בנוסף, ניתן לבדוק את חוקי ה-RLS והאבטחה של Supabase על ידי הרצת:
```bash
npm run check:rls
```

## 4. מדריך פריסה לענן (Production Deployment)

האתר מותאם לפריסה מהירה ורציפה (CI/CD) בפלטפורמות כדוגמת **Netlify** או **Vercel** ומחובר ישירות לפרויקט Supabase בענן.

### הגדרות משתני סביבה (Environment Variables) בשרת הייצור
בממשק הניהול של פלטפורמת הפריסה (למשל ב-Netlify Environment Variables), יש להגדיר את המשתנים הבאים:

| שם המשתנה | תיאור | ערך לדוגמה |
| :--- | :--- | :--- |
| `NEXT_PUBLIC_SUPABASE_URL` | כתובת פרויקט ה-Supabase שלכם בענן | `https://your-project.supabase.co` |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | מפתח Anon של פרויקט ה-Supabase | `eyJhbGciOiJIUzI1...` |
| `SUPABASE_SERVICE_ROLE_KEY` | מפתח Service Role סודי (לא נחשף לדפדפן!) | `eyJhbGciOiJIUzI1...` |
| `RESEND_API_KEY` | מפתח אימות לשליחת מיילים דרך שירות Resend | `re_123456789...` |
| `NEXT_PUBLIC_SITE_URL` | הכתובת הציבורית של האתר שלכם | `https://your-domain.co.il` |
| `NEXT_PUBLIC_CONTACT_EMAIL` | כתובת אימייל ליצירת קשר ותמיכה משפטית (נגישות/פרטיות) | `contact@your-domain.co.il` |

### פריסה רציפה באמצעות GitHub
1. חברו את מאגר הקוד (Repository) ב-GitHub ל-Netlify/Vercel.
2. הגדירו את פקודת הבנייה (Build command): `npm run build`.
3. הגדירו את תיקיית הפלט (Publish directory): `.next`.
4. בכל ביצוע Push לענף הראשי (`main`), שרת הפריסה יבנה ויפרוס את האתר מחדש באופן אוטומטי תוך הרצת בדיקות האיכות.

## 5. צ'קליסט אבטחה והשקה (Security Launch Checklist)

לפני עלייה מלאה לאוויר, יש לבצע את השלבים הבאים **יחד עם בעל האתר** (אין לבקש או להחזיק בפרטי הגישה האישיים שלו לחשבונות השונים):

### שלב א': החלת מיגרוציות אבטחה ב-Supabase Production
ודאו שכל מיגרוציות בסיס הנתונים (עד `0009_storage_insert_policy_fix.sql`) מוחלות במלואן על פרויקט הייצור בענן:
```bash
npx supabase db push --db-url "postgres://postgres:[YOUR-PROD-PASSWORD]@aws-0-eu-central-1.pooler.supabase.com:6543/postgres"
```

### שלב ב': הגדרות אימות ב-Supabase Auth
היכנסו לפורטל הניהול של Supabase -> Auth -> Providers:
1. **Email Confirmations:** הפעילו חובת אישור אימייל (`Confirm Email`) בעת הרשמה כדי למנוע הרשמות של כתובות פיקטיביות.
2. **Password Policy:** הגדירו אורך סיסמה מינימלי של 8 תווים לפחות והפעילו הגנת סיסמאות דלופות (Leaked passwords protection).
3. **Redirect URLs:** הגדירו את רשימת הכתובות המורשות להפניה (Redirect URLs) כך שיכילו אך ורק את הדומיין הרשמי של האתר בייצור (לדוגמה `https://your-domain.co.il/**`) ומחקו כתובות בדיקה זמניות.

### שלב ג': הגנת בוטים וספאם (CAPTCHA)
1. היכנסו ל-Supabase -> Auth -> Security:
2. הפעילו הגנת CAPTCHA (כגון Cloudflare Turnstile או hCaptcha) על טפסי ההרשמה הציבוריים כדי למנוע יצירת משתמשים פיקטיביים אוטומטית.

### שלב ד': אבטחת חשבונות מנהל (MFA)
1. כל מנהל מערכת (Admin) בעל גישה לפאנל הניהול `/admin` נדרש להפעיל אימות דו-שלבי (MFA/2FA) בחשבונו בתוך האתר/הדשבורד כדי להגן על מידע המאמצים הרגיש במקרה של גניבת סיסמה.

### יצירת מנהל המערכת הראשון בייצור

סקריפט ה-`bootstrap-admin.mjs` מוגן ב-localhost ולא ירוץ מול ייצור (בכוונה). עדכון ישיר של
`role` ב-SQL Editor ייחסם על ידי טריגר האבטחה `guard_profile_privileges`. הדרך הנכונה:

1. מנהלת האתר נרשמת באופן רגיל באתר החי (`/signup`) עם כתובת האימייל האמיתית שלה וסיסמה חזקה (ומאשרת את האימייל, אם אישור אימייל כבר הופעל).
2. ב-SQL Editor של פרויקט הייצור מריצים את הטרנזקציה הבאה (הצהרת service_role עוברת את הטריגר):

```sql
begin;
select set_config('request.jwt.claims', '{"role":"service_role"}', true);
update public.profiles
  set role = 'admin'
  where id = (select id from auth.users where email = 'admin@example.co.il'); -- החליפו לאימייל האמיתי
commit;
```

3. התנתקות והתחברות מחדש — האפשרות **איזור מנהל** תופיע בניווט ותוביל ל-`/admin`.
4. מפעילים MFA לחשבון (שלב ד' לעיל) מיד לאחר מכן.

### שלב ה': הגדרת אימות דוא"ל ב-Resend
1. היכנסו לחשבון ה-Resend של בעל האתר.
2. הגדירו את הדומיין הייצורי תחת `Domains` והוסיפו את רשומות ה-DNS הנדרשות (רשומות **SPF** ו-**DKIM**) אצל רשם הדומיינים שלו על מנת להבטיח מעבר תקין של המיילים לתיבת הדואר הנכנס (Inbox) ומניעת הגעתם לספאם.


### שלב ו': אימותים לאחר העלייה לאוויר (Post-Deploy)
1. **Lighthouse בסביבת הייצור:** הריצו בדיקת Lighthouse (מובייל) על עמוד הבית ועמוד הקטלוג בכתובת הייצור. היעד: ציון Performance ‏90 ומעלה בשני העמודים (החלטת ארכיטקט, 2026-07-16: האימות הפורמלי של היעד מתבצע מול סביבת הייצור). אם הציון נמוך מ-90 — פותחים סבב אופטימיזציה.
2. **בדיקת תמונות שיתוף (OG):** ודאו שכתובת `og:image` של עמוד חתול נטענת ללא עוגיות (למשל דרך כלי ה-Sharing Debugger של פייסבוק).
3. **רישיונות מדיה של ההירו:** אתרו מחדש את עמודי המקור המדויקים של 4 קליפי ההירו ב-Mixkit ועדכנו את הקישורים ב-`docs/hero-media-licenses.md` (כיום מסומנים "not recorded").
