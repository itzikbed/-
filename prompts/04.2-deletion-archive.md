Background you must know: `public.cats` currently has NO delete policy, so
`deleteCatAction` (`app/publish/cat-actions.ts:112`) purges the cat's photos/video from
storage, then the row delete silently affects 0 rows, and the action still returns
`ok:true` — a published listing survives with dead images. Item 2 fixes this.

## Item 1 — migration 0007: owner delete policy

New file `supabase/migrations/0007_cats_delete_policy.sql` (never edit applied migrations):

```sql
create policy cats_delete_owner on public.cats for delete
  using (
    owner_id = auth.uid()
    and published_at is null
    and status in ('draft','pending','rejected')
  );
```

No delete policy for admins — per §5a, admin removal is archive, and the two hard-delete
exceptions (privacy purge, unlawful content) run as service role outside the app.
No table changes ⇒ no `database.types.ts` regeneration needed.

## Item 2 — fix `deleteCatAction`: row first, verify, then storage

Rewrite the ordering in `app/publish/cat-actions.ts`:

1. Load the cat (need `published_at`, `video_path`, photo paths).
2. If `published_at IS NOT NULL` ⇒ return `{ok:false}` with a Hebrew message directing to
   archive instead ("מודעה שפורסמה אינה נמחקת — ניתן להעביר לארכיון").
3. Delete the DB row with `.delete().eq('id', …).select()` and require exactly one
   returned row; 0 rows ⇒ `{ok:false}` ("לא ניתן למחוק את המודעה") — do NOT touch storage.
4. Only after the row is confirmed gone: remove storage files (photos + video). A storage
   failure here is non-fatal: log it, still return `ok:true` (orphan files are a cleanup
   concern, not a user error).

Keep the admin role-check branch out: after this change the action is owner-only (RLS
enforces it anyway); admins never hard-delete via the UI.

## Item 4 — admin removal control (archive with reason)

Admins need a way to remove an already-published listing (§5a). Lean approach — NO fourth
queue: on `/cats/[id]`, render an admin-only control ("הסרה מהאתר") when the viewer's
role is admin. It opens the existing reject-style dialog (reason mandatory, ≥10 chars) and
calls a new `archiveCatAdminAction(catId, reason)` in `app/admin/cat-actions.ts`:

- `checkAdmin()`, guarded update `published`→`archived` (strict `.eq('status','published')`,
  0 rows ⇒ conflict message), `moderation_log` insert `action='archive'` + reason,
  fire-and-log email to the owner (new template, Item 6), sibling auto-close (Item 5),
  revalidate `/cats`, `/cats/[id]`, `/admin`, `/`.
- Note: the public page currently 404s non-published cats for everyone — make sure the
  admin (and owner) path can still view a published cat normally; the control only needs
  to appear while the cat IS published, so no visibility changes are required.

## Item 5 — sibling auto-close on leaving `published`

Shared helper (e.g. `lib/requests/close-siblings.ts`, server-only): given a `catId` and the
service-role client (`createAdminClient()` — RLS does not allow owners to touch other
users' requests), set all that cat's `adoption_requests` with `status='pending'` to
`status='rejected'`, `admin_note` = auto text from `lib/strings.ts` (gender-aware:
"‏{name} כבר נמסר/ה לאימוץ"), `decided_at=now()`; `decided_by` stays null (system).
Then fire-and-log an email per closed request (Item 6 template). Failures are logged,
never fatal to the caller's decision (same pattern as the existing email tails).

Call it from: `markAsAdoptedAction`, `archiveCatAction`, `archiveCatAdminAction` — only
after their guarded update actually matched a row.

Approving one request must NOT close siblings (contact handoff ≠ final adoption) — do not
"improve" this.
